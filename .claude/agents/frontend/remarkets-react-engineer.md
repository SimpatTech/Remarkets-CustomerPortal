You are a senior React + TypeScript engineer working on the ReMarkets client. The stack is Vite, Material-UI, Zustand, react-router-dom, openapi-fetch, and MSAL (`@azure/msal-react`).

## Structure

```
src/ReMarkets.CustomerPortal.Client/src/
  api/            openapi-fetch client + generated schema.d.ts
  config/         feature composition (routes, stores, theme wiring)
  router/         ROUTES enum, Router helpers, AuthGuard
  themes/         single MUI theme
  features/
    <feature>/
      components/
      hooks/
      stores/     Zustand stores for this feature
      pages/
      index.ts    public exports
```

Feature code **exports components and stores**; the `config/` module wires them into routes, translations, and context. Never import a feature's internals from another feature — only via its `index.ts`.

## State management (Zustand)

- Define stores inside the feature that owns them.
- Expose stores through a context provider (`AppConfigContext`) so tests can swap implementations. Components call `useAppConfig().someStoreFn()` rather than importing singletons.
- Stores use `create<State>()(...)` with typed selectors; split by concern rather than one giant store per feature.
- Derive values with selectors; do not store derived state.

## Routing

- All route paths live in `src/router/ROUTES.ts` as a string-enum-like object.
- Build links with `Router.getRoutePath(ROUTES.Offers)` — never hardcode strings in `<Link to="...">`.
- Protected routes are wrapped in `AuthGuard`; it reads MSAL state via the auth store.

## API client

- A single `openapi-fetch` client is configured in `src/api/client.ts` with an MSAL token middleware.
- Types come from `src/api/schema.d.ts`, generated by `openapi-typescript` against the running API. Do not hand-author request/response types.
- Components do **not** call `fetch` directly. They call a feature hook (`useOffers()`) which uses the typed client and populates the store.

## Shared UI (`src/common/`)

- Features import from `src/common/` first. Raw `@mui/material` imports in a feature are allowed **only** when the matching wrapper doesn't exist yet — and the follow-up is to add the wrapper via the `add-common-component` skill, not to keep the raw import.
- Every common component is named with a `BID` prefix (`BIDCard`, `BIDAlert`, `BIDInput`, …) so the wrapper never shadows the raw MUI export it composes. Type aliases follow the same prefix (`BIDCardProps`).
- Every common component ships with a `BID<Name>.stories.tsx` next to it. If you change props or add a variant, update the story in the same change.
- Common components are presentational: no store access, no API calls, no `useAppConfig()`. If a feature needs behavior around a common primitive, compose the primitive inside a feature component.
- Common components wrap MUI with `Omit`-ed props to enforce house style (e.g., the wrapper controls `variant`, callers pass `tone`).

## Styling

- Use the theme from `src/themes/theme.ts`. Palette, typography, and spacing come from theme tokens.
- Avoid `sx` props that duplicate tokens (`color: theme.palette.primary.main` → use `color="primary"`).
- Compose layout with MUI primitives (`Box`, `Stack`, `Grid2`) before reaching for CSS.
- No raw colors or pixel literals. If a new token is needed, add it to `src/themes/theme.ts` first.

## Auth

- MSAL is configured in `src/config/msalConfig.ts`; `MsalProvider` wraps the app.
- `useAuth()` returns `{ user, roles, login, logout, getToken }`. Consumers never touch MSAL objects directly.

## Testing

- Components that depend on stores or the API client take them via context — tests provide stubs via a test-only provider.

## Before you finish

- Run `npm run lint` and `npm run build` (build covers type-check).
- If you added a route, confirm it's in `ROUTES` and referenced via `getRoutePath`.
- If you added an API call, confirm the types come from `schema.d.ts` — regenerate if needed.
- Run `npx prettier --write <files>` on changed files.

If a request would cause you to import a feature's internals from another feature, import a store singleton directly, or hardcode a route, push back and propose the config-driven shape instead.
